# DSA 短提示词旁路（Short-Prompt Bypass）代码检视指南

##### https://github.com/licheng165/vllm/commit/3fec8b46cfe3aa0fc91fb917bb22df7e9d2e7fad
##### https://github.com/licheng165/vllm-ascend/commit/d72ae5a6c3b8b13f62a4b2fb1ed081f6d2589719
##### https://github.com/licheng165/LMCache/commit/1f358235f04162adc559611520964b3f1c0aef33

## 一、背景

### 1.1 DSA 特性回顾

DSA（Dynamic Sparse Attention）是面向 MLA 架构模型（GLM-5.1、DeepSeek-V3）的显存优化方案。核心思想是"稀疏注意力不仅稀疏计算，也稀疏存储"：

- **Prefill 阶段**：计算全量 latent KV，保存到 LMCache（CPU 内存），然后释放 NPU 上的 latent 块，只保留 scratch（top-k 个 token 的空间）
- **Decode 阶段**：indexer 选出 top-2048 个最相关 token，从 LMCache 按需加载这 2048 个 token 的 latent KV 到 scratch，做稀疏注意力

### 1.2 问题

DSA 特性**仅对长提示词有收益**。当 `prompt_len <= scratch_size` 时：

1. **无内存节省**：`DSALatentManager.remove_skipped_blocks` 中 `end <= start`，不释放任何块
2. **纯开销**：LMCache save（Prefill 时 H2D 传输到 CPU）+ scratch_remap（每 decode 步重映射 + 选择性 H2D 加载）照常执行
3. **正确性风险**：已有代码 `model_runner_v1.py:2310` 警告"compact-scratch path does not support it yet... Expect wrong output"

### 1.3 scratch_size 与 MTP 的关系

scratch 大小取决于 MTP 的 `num_speculative_tokens`：

```
scratch_blocks = ceil(index_topk * (1 + num_speculative_tokens) / block_size)
scratch_size   = scratch_blocks * block_size
```

| num_speculative_tokens | decode_threshold | scratch_size (block_size=128, index_topk=2048) |
|------------------------|------------------|-------------------------------------------------|
| 0（无 MTP）            | 1                | 2048                                            |
| 1                      | 2                | 4096                                            |
| 3                      | 4                | 8192                                            |

每个 MTP 投机 token 行各需 `index_topk` 个 scratch slot，因此阈值必须随 `num_speculative_tokens` 缩放。

## 二、修改方案概述

**核心思路**：当 `prompt_len <= scratch_size` 时，让请求走非稀疏路径（original code branch），跳过 LMCache offload/load 和 scratch_remap。

**阈值**：`scratch_blocks * block_size`（纯推导值，无环境变量），由模型的 `index_topk`、MTP 的 `num_speculative_tokens` 和 `block_size` 唯一确定。

### 2.1 三个拦截点

| # | 仓库 | 文件 | 拦截点 | 效果 |
|---|------|------|--------|------|
| 1 | vllm-ascend | `sfa_v1.py` | `split_boundary = 0`（短提示词 decode 行） | SFA 跳过 scratch_remap 和选择性 LMCache 加载，走 native 路径直接读 NPU 常驻 KV |
| 2 | LMCache | `vllm_v1_adapter.py` | `is_sparse_decode = False`（短提示词） | LMCache 走普通 layerwise 路径（全量加载），不走稀疏选择性加载 |
| 3 | vllm | `single_type_kv_cache_manager.py` | `remove_skipped_blocks` 提前返回 | 不释放 latent 块，KV 留在 NPU 显存 |

### 2.2 单机 vs PD 分离的区分

通过 `disagg_spec` 自动区分，无需额外配置：

| 场景 | `disagg_spec` | skip_save 行为 | LMCache save | LMCache load |
|------|---------------|----------------|--------------|--------------|
| 单机 8 卡 | `None` | `True`（跳过保存） | 不保存 | 不加载（`load_spec=None` → `from_request_tracker` 返回 `None`） |
| PD 分离 Prefiller | `非 None` | `False`（保留保存） | 保存到 mooncake | 不加载 |
| PD 分离 Decoder | `None` | `True`（跳过保存） | 不保存 | 从 mooncake 全量加载（`load_spec.can_load=True`） |

**关键安全机制**：`from_request_tracker` 第 995 行 `if skip_save and load_spec is None: return None`
- 单机短提示词：`skip_save=True` + `load_spec=None` → 返回 `None` → 完全无 LMCache 交互
- PD Decoder 短提示词：`skip_save=True` + `load_spec!=None`（mooncake 有 KV）→ 继续 → 加载但不保存

### 2.3 MTP 适配

- `split_boundary=0` 是 per-row 设置，`req_rows` 和 `row_offsets` 仍正常设置 → MTP 不相交 scratch base 正确工作
- `is_sparse_decode=False` 是 per-request 设置 → MTP 投机 token 由 SFA native 路径（`SpecDecoding`）处理
- 阈值随 `num_speculative_tokens` 自动缩放

## 三、逐文件修改详情

### 3.1 vllm-ascend: `vllm_ascend/attention/sfa_v1.py`

#### 修改 A：新增 `_compute_short_prompt_threshold` 方法

**位置**：`AscendSFAMetadataBuilder` 类，`determine_chunked_prefill_workspace_size` 之后

```python
def _compute_short_prompt_threshold(self) -> int:
    """Short-prompt threshold: prompts with prompt_len <= this value
    bypass the DSA sparse-decode path.

    The threshold is the DSA scratch size, derived from the model config
    and MTP configuration:

        scratch_blocks = ceil(index_topk * decode_threshold / block_size)
        threshold      = scratch_blocks * block_size

    where decode_threshold = 1 + num_speculative_tokens.  When
    prompt_len <= scratch_size, no latent blocks are freed (end <= start
    in DSALatentManager.remove_skipped_blocks), so the LMCache save/load
    and scratch_remap are pure overhead with zero memory savings.
    """
    scratch_blocks = (
        self.index_topk * self.decode_threshold + self.block_size - 1
    ) // self.block_size
    return scratch_blocks * self.block_size
```

**`__init__` 中调用**（`self._dsa_target_position_offsets` 赋值之后）：
```python
self._dsa_short_prompt_threshold = self._compute_short_prompt_threshold()
```

**检视要点**：
- `self.index_topk`、`self.decode_threshold`、`self.block_size` 均在 `__init__` 前面已赋值（line 333、337-340、357-363）
- `decode_threshold = 1 + num_speculative_tokens`（line 337-340），MTP 未配置时为 1
- 公式与 `vllm/v1/core/dsa_shared_pool.py:dsa_scratch_blocks_for_topk` 一致

#### 修改 B：`build()` 方法中设置 `split_boundary=0`

**位置**：`AscendSFAMetadataBuilder.build()`，per-request 循环内（原 line 463）

```python

# 修改后

    rows[first_decode:e] = 0
```

**检视要点**：
- `rows` 即 `split_boundary_rows`，控制 `prepare_sparse_indices` 的重映射行为
- `split_boundary=0` 意味着所有位置 >= 0，无 LMCache-owned 位置 → `_sel_packed` 为空 → 不触发 LMCache 加载
- `req_rows[first_decode:e] = r` 和 `row_offsets[first_decode:e] = offsets` 不变 → decode 行计数和 MTP scratch base 仍正确
- 混合 batch（短+长）正确：per-row 设置，短提示词行为 0、长提示词行为 plen

## 四、场景验证清单

### 4.1 单机 8 卡场景

| 步骤 | 短提示词行为 | 长提示词行为（不变） |
|------|-------------|---------------------|
| 新请求 `build_connector_meta` | `skip_save=True`（无 disagg_spec） | `skip_save=False` |
| Prefill SFA forward | `save_spec.can_save=False` → 不保存到 LMCache | 保存 latent 到 LMCache |
| Prefill 结束 `remove_skipped_blocks` | `prompt_len <= scratch_size` → 提前返回，不释放块 | 释放 `[scratch..prompt)` 范围的 latent 块 |
| Decode `build_connector_meta` | `is_sparse_decode=False` + `skip_save=True` | `is_sparse_decode=True` |
| Decode `start_load_kv` | `load_spec=None` → `from_request_tracker` 返回 `None` → 无 ReqMeta → 无 LMCache 交互 | 走稀疏选择性加载路径 |
| Decode SFA forward | `split_boundary=0` → `_sel_packed=None` → 不加载 → native 路径读 NPU 常驻 KV | scratch_remap + 选择性 LMCache 加载 |

### 4.2 四机 PD 分离场景

| 步骤 | Prefiller（短提示词） | Decoder（短提示词） |
|------|----------------------|---------------------|
| 新请求 `build_connector_meta` | `req_id in tmp_disagg_tracker` → `skip_save=False` | `req_id not in tmp_disagg_tracker` → `skip_save=True` |
| Prefill SFA forward | `can_save=True` → **保存到 mooncake** | N/A（Decoder 不做 prefill） |
| Prefill 结束 | `prompt_len <= scratch_size` → 不释放块 | N/A |
| 迁移 | KV 通过 mooncake 传输到 Decoder | 接收请求，`update_state_after_alloc` 检查 mooncake → `load_spec.can_load=True` |
| Decode `build_connector_meta` | N/A（请求已迁移） | `is_sparse_decode=False` + `skip_save=True`（`disagg_spec=None`） |
| Decode `start_load_kv` | N/A | `load_spec!=None` → `from_request_tracker` 继续 → **从 mooncake 全量加载** |
| Decode SFA forward | N/A | `split_boundary=0` → native 路径读 NPU 常驻 KV |

### 4.3 MTP 场景

| `num_speculative_tokens` | 阈值 | `split_boundary=0` | MTP scratch base | SFA native 路径 |
|--------------------------|------|---------------------|------------------|-----------------|

### 4.4 混合 batch 场景

| 维度 | 短提示词请求 | 长提示词请求 |

| SFA `split_boundary` | 该行为 0 | 该行为 plen |

## 五、举一反三：同类问题修复

审计发现两处与短提示词阈值同源的问题——固定使用 `index_topk`（2048）而非 MTP 感知的 `index_topk * decode_threshold`：

### 5.1 model_runner 警告阈值

**问题**：`model_runner_v1.py:2313` 用 `self.dsa_index_topk`（2048）做比较，MTP=3 时应为 8192。一个 3000 token 的提示词不触发警告（3000 > 2048）但实际被旁路（3000 < 8192）。

**修复**：改用 `(self.dsa_index_topk * self.decode_threshold + self.block_size - 1) // self.block_size * self.block_size`。

### 5.2 `SPARSE_DECODE_RETRIEVE_TOKENS` slot mapping 上限

**问题**：`from_request_tracker` 用 `_sparse_slot_mapping_len`（上限 2048 = `index_topk`）限制 `tracker.sparse_slot_mapping` 大小。MTP=1 时需 4096 个 slot，上限不足。

**修复**：新增 `sparse_slot_cap` 参数，`build_connector_meta` 三处调用均传入 `self._dsa_short_prompt_threshold`（= `scratch_blocks * block_size`，MTP 感知）。

### 5.3 共同根因

两处都是用 **per-row 值**（`index_topk`）代替了 **per-request 值**（`index_topk * decode_threshold`）。MTP 的每个投机 token 行各需 `index_topk` 个 scratch slot，`decode_threshold` 行总共需要 `index_topk * decode_threshold` 个。

## 六、检视检查清单

### 6.1 正确性

- [ ] 阈值公式 `ceil(index_topk * decode_threshold / block_size) * block_size` 在三处实现一致（SFA、LMCache adapter、DSALatentManager）
- [ ] 单机场景：短提示词完全不与 LMCache 交互（`from_request_tracker` 返回 `None`）
- [ ] PD 分离 Prefiller：短提示词仍保存到 mooncake（`disagg_spec` 使 `skip_save=False`）
- [ ] PD 分离 Decoder：短提示词从 mooncake 全量加载（`load_spec!=None` 使 `from_request_tracker` 不返回 `None`）
- [ ] 混合 batch：per-row `split_boundary` 和 per-request `is_sparse_decode` 独立工作
- [ ] MTP：`req_rows` 和 `row_offsets` 不受 `split_boundary=0` 影响
- [ ] Preemption：`skip_save=True` 持久化，`prompt_len` 不可变，行为一致
- [ ] Chunked prefill：`split_boundary` 只在 decode 行设置，prefill 行不受影响
- [ ] Decode window save：`_decode_window_save_window_size > 0` 检查在 short-prompt 检查之前（`DSALatentManager`），decode window save 不受 `skip_save` 影响（独立 `ReqMeta`）

### 6.2 性能

- [ ] 短提示词无 LMCache H2D 传输开销（单机）
- [ ] 短提示词无 scratch_remap 开销（`split_boundary=0` → `prepare_sparse_indices` 无重映射）
- [ ] 短提示词无 latent 块释放开销（`remove_skipped_blocks` 提前返回）
- [ ] PD 分离 Decoder 短提示词一次性全量加载（而非每 decode 步选择性加载）

### 6.3 代码风格
